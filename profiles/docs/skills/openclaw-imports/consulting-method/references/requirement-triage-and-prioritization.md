# 需求分诊与优先级框架

## 目标
当用户一次性给出大量需求、问题清单、访谈纪要、需求池时，帮助 consulting agent 快速完成分诊、分类、排序，而不是平铺直叙地复述。

## 一、先做四分法分诊
所有需求优先分入 4 类：

### A. 紧急修复类
特征：影响业务连续性、存在系统缺陷、接口异常、数据错误、关键流程中断。
例子：WMS 故障、ATP 缺陷、过磅系统异常、接口不同步。

### B. 闭环打通类
特征：端到端流程不完整，跨系统断裂，主系统不唯一。
例子：SRM-SAP-OA 未闭环、CRM 到订单回款断裂、费控到 ERP/资金未打通。

### C. 分析提效类
特征：日报、月报、BI、经营看板、自动分析、预警。
例子：采购日报自动生成、库存报告自动生成、物流 BI 驾驶舱。

### D. 能力建设类
特征：AI、知识库、预测、情报抓取、可视化、创新性工具。
例子：趋势分析 AI、3D 展示、竞品爬虫、路线优化算法。

## 二、默认优先级原则
通常优先顺序：
1. 紧急修复类
2. 闭环打通类
3. 分析提效类
4. 能力建设类

## 三、判断优先级的五个维度
每个需求都至少判断：
1. 业务影响：是否直接影响收入、交付、回款、采购、生产、付款
2. 风险程度：是否影响合规、数据准确、业务连续性
3. 跨部门程度：是否牵涉多个部门，是否属于主流程关键节点
4. 前置依赖：是否依赖主数据、流程标准化、系统接口先完成
5. 落地难度：是否具备一期快速闭环条件

## 四、分诊后的输出方式
面对大量需求时，默认输出：
1. 总体判断：这批需求的问题结构是什么
2. 分类结果：按四分法归类
3. P1/P2/P3 建议：哪些先做，为什么
4. 一期建议：推荐做哪些
5. 暂缓建议：哪些不建议当前就做

## 五、识别“伪高优先级”
以下需求经常声音很大，但不一定该先做：
- 可视化很好看，但底层数据不稳
- AI 功能新颖，但主流程还没闭环
- 局部提效明显，但不会改善整体经营结果
- 只有个别部门强烈想要，但集团级价值不高

## 六、识别“真高优先级”
以下通常应优先：
- 直接影响核心交易与经营闭环
- 直接影响财务准确性或资金安全
- 影响多个部门协同
- 会阻碍后续更多项目开展
- 修复后能为后续分析/AI 能力打基础

## 七、常用结论表达
- 这批需求的核心矛盾不是功能不够，而是主流程闭环和数据治理基础薄弱。
- 当前不建议把资源优先投向 AI/BI 展示层，应该先修复关键业务链路。
- 一期应聚焦影响主流程闭环的少数关键需求，而不是全面铺开。
