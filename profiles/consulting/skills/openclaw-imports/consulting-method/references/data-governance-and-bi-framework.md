# 数据治理与经营分析框架

## 目标
帮助 consulting agent 在面对“数据、报表、BI、分析、AI 自动生成报表、主数据治理”等问题时，不只停留在报表层，而是能判断数据治理成熟度、分析价值和实施路径。

## 一、先判断这是不是假 BI 需求
很多“我要 BI / 仪表盘 / AI 报表”的真实诉求，其实不是 BI 问题，而是：
- 源系统断裂
- 主数据不统一
- 指标口径不一致
- 数据采集不完整
- 业务动作没有闭环

默认先问四件事：
1. 这个报表/分析要支持什么经营动作？
2. 数据来自哪些源系统？
3. 指标口径是谁定的？
4. 问题是看不见，还是本来就没管好？

## 二、数据问题的五层结构

### L1 源数据层
看的是：数据有没有被正确采集。
常见问题：系统外 Excel、线下补录、关键字段缺失、状态没回写。

### L2 主数据层
看的是：客户、供应商、物料、项目、组织、仓库、账户等是否统一。
常见问题：多头维护、编码混乱、归属不清。

### L3 交易数据层
看的是：订单、收货、发货、报销、付款、回款等业务单据是否形成稳定链路。
常见问题：单据断裂、来源不唯一、跨系统状态不一致。

### L4 指标口径层
看的是：收入、成本、毛利、库存、逾期、预算执行等是否统一定义。
常见问题：财务口径和业务口径打架，不同部门报表互相对不上。

### L5 经营分析层
看的是：分析能否支持预警、决策、考核、优化。
常见问题：图很多，但不能驱动行动。

原则：
- L1-L4 不稳，L5 只能做展示，不能做管理。

## 三、主数据治理默认判断框架
### 重点对象
- 客户
- 供应商
- 物料
- 项目
- 组织
- 仓库/地点
- 科目/费用项
- 银行账户/资金主体

### 要判断的 5 件事
1. 谁创建
2. 谁审批
3. 谁维护
4. 谁消费
5. 谁对质量负责

如果这 5 件事说不清，后面的报表通常都不稳。

## 四、报表需求的顾问拆法
用户说“要一个报表”时，默认拆成：
1. 用途：看什么问题
2. 对象：看哪个业务域
3. 粒度：按单据、客户、物料、项目，还是组织
4. 频率：实时、日、周、月
5. 动作：看完之后谁要做什么动作

如果一个报表没有明确动作对象，它通常只是展示需求，不是经营需求。

## 五、BI / 驾驶舱建设的正确顺序
### 第一步：先统一指标口径
明确指标定义、统计范围、时间口径、组织口径。

### 第二步：梳理源系统与字段
不要一上来建大宽表，先确认源头和映射。

### 第三步：补关键缺失字段和状态回写
有些报表做不出来，不是因为没工具，而是源系统根本没记。

### 第四步：分主题域建设
常见主题域：
- 销售与回款
- 采购与供应商
- 库存与周转
- 费用与预算
- 项目经营
- 资金与现金流
- 物流与履约

### 第五步：先做管理闭环看板
优先做能驱动经营动作的报表，不要先堆大而全驾驶舱。

## 六、经营分析常见误区
### 误区 1：把自动化出报表当成经营改善
自动化只是提效，不等于管理提升。

### 误区 2：报表越多越好
真正有效的是少数关键指标 + 明确责任动作。

### 误区 3：只看结果，不看过程
只看销售额、库存额、费用额，无法定位问题；要看过程链路指标。

### 误区 4：先上 BI，再补基础数据
顺序反了，后面必然返工。

## 七、咨询时的默认输出结构
面对数据/BI类问题，尽量输出：
1. 结论：这是治理问题、口径问题，还是展示问题
2. 当前短板：源数据 / 主数据 / 交易链路 / 指标口径 / 分析动作
3. 优先动作：先补什么
4. 一期建议：先做哪个主题域
5. 风险与前提：谁必须参与、哪些定义必须先统一

## 八、与 AI 需求结合时的判断
如果用户提“AI 自动生成日报、AI 分析经营、AI 预测”，要先判断：
- 数据源是否稳定
- 指标口径是否可复用
- 输出是否要进入实际经营动作
- 是否只是把人工汇总改成 AI 汇总

如果基础数据不稳，AI 只会更快地产生不可信结果。
