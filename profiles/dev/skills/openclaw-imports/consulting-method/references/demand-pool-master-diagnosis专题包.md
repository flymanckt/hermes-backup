# 需求池总诊断与优先级总图专题包

## 目标
用于帮助 consulting agent 面对大批量需求、项目池、信息化建设清单、老板拍来的需求合集时，能够做组合管理和优先级判断，而不是简单做线性排序。

## 一、默认结论框架
当用户提到以下场景时：
- 需求很多，不知道先做什么
- 老板、业务、IT 各有一堆需求
- 今年项目太多，资源不够
- 想做整体优先级排序
- 要做数字化蓝图和年度项目计划

优先默认判断：
1. 需求池不是待办列表，而是资源投资组合。
2. 真正要先排的不是“谁声音大”，而是“谁影响业务连续性、主流程闭环和经营透明度”。
3. 没有分类和前提判断，直接全量打分通常会失真。

## 二、需求池四类组合

### 1. 稳定运行组合
目的：保连续性、保准确性、保系统不掉链子。
典型：系统缺陷、接口故障、性能问题、运行异常、关键报表错误。

### 2. 主流程闭环组合
目的：打通采购、销售、生产、库存、费用、资金等主链路。
典型：P2P/O2C/E2R/制造计划库存/项目经营闭环。

### 3. 管理提效组合
目的：提升效率、透明度、日常协同和报表自动化。
典型：日报、周报、BI、审批提效、自动提醒、电子签章。

### 4. 能力建设组合
目的：建立中长期能力与创新空间。
典型：AI、预测模型、知识库、情报系统、可视化创新、复杂优化。

## 三、默认排序原则
### 原则 1：先保运行，再谈创新
底层不稳时，创新项目很容易返工。

### 原则 2：先闭环，再提效
主流程没闭环，自动化和分析的收益会被打折。

### 原则 3：先数据基础，再智能化
主数据、口径、源数据不稳时，AI/BI 只能放大噪音。

### 原则 4：先抓项目群，再抓零散需求
企业项目管理应该按主题项目群推进，而不是零碎功能逐条推进。

## 四、典型诊断步骤
1. 先看需求总体结构分布
2. 再识别当前企业主矛盾
3. 再确认哪些需求存在依赖关系
4. 再划项目群
5. 最后给优先级与分期建议

## 五、默认诊断维度
### 1. 业务影响
是否影响收入、交付、成本、现金流、客户体验。

### 2. 管理价值
是否增强透明度、控制力、跨部门协同。

### 3. 紧急程度
是否影响业务连续性或关键节点。

### 4. 依赖关系
是否依赖主数据、系统集成、流程标准化、其他项目先完成。

### 5. 实施复杂度
跨部门程度、系统数量、数据难度、组织变更难度。

### 6. 可落地性
有没有 owner、有没有时间窗口、业务愿不愿配合。

## 六、建议输出方式

### 1. 总体结构判断
先说：当前需求池的主矛盾是什么。

### 2. 分类结果
按四类组合拆解需求，不要直接堆表。

### 3. 当前优先投资的项目群
建议 3 个以内，不要过多。

### 4. 分期建议
- 一期先做
- 二期再做
- 暂缓事项

### 5. 风险与前提
- 需要谁做 owner
- 需要哪些基础前提先解决

## 七、优先级总图的默认表达
建议形成：
- 高价值 / 低复杂度：尽快做
- 高价值 / 高复杂度：列为核心项目群
- 低价值 / 低复杂度：择机做
- 低价值 / 高复杂度：谨慎做

## 八、默认追问清单
1. 这些需求是同一个年度项目池，还是不同阶段混在一起？
2. 现在最影响经营或管理的 3 个问题是什么？
3. 哪些需求如果不做，会直接影响业务运行？
4. 哪些需求强依赖主数据、系统集成或流程标准化？
5. 今年真实能投入多少项目资源？

## 九、默认输出结构
1. 结论
2. 需求池结构判断
3. 项目群划分
4. 优先级建议
5. 分期建议
6. 风险与前提

## 十、不要犯的错误
- 不要把需求池当成简单待办清单。
- 不要直接按部门声音大小排序。
- 不要忽略依赖关系与前置条件。
- 不要一期同时开太多项目群。
- 不要把 BI/AI 项目放在基础治理前面。
