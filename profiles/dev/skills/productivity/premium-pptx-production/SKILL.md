# Premium PPTX Production

## 何时触发

当用户提到以下任一需求时，必须触发本技能：

- 生成 PPT / 演示文稿 / slides / deck
- 输出 `.pptx`
- 要求“高级感”“像 Kimi 一样”“像专业咨询公司/投行/管理汇报一样”
- 需要电脑端 PowerPoint / WPS 正常打开
- 需要给客户、管理层、董事会、项目汇报、方案汇报、路演、售前方案使用

如果用户只是要 HTML 演示页，不用本技能；如果需要 `.pptx`，必须优先本技能。

---

## 核心目标

本技能不是“做一份能看 PPT”，而是做一份满足以下标准的 **高质量可交付 `.pptx`**：

1. **可打开性稳定**
   - Windows PowerPoint 打开不报修复
   - WPS / LibreOffice 基本兼容
   - 文件结构完整
2. **视觉高级感**
   - 有明确视觉主线，不是普通白底黑字 bullet 文档
   - 具备管理汇报、咨询汇报、科技方案汇报气质
3. **内容表达专业**
   - 页面标题结论化
   - 一页一结论
   - 信息密度适中
4. **可复用、可落地**
   - 优先使用稳定生成链路
   - 必须做生成后 QA，而不是生成完直接交付

## 质量底线（强制）

任何 agent 使用本技能时，都必须遵守以下硬性规则：

### A. 文件规则
- 最终输出必须为 `.pptx`
- 禁止只给 markdown / html / 截图替代 `.pptx`
- 必须给出最终输出路径

### B. 打开规则
- 生成后必须做一次**结构可打开性验证**
- 至少完成以下之一：
  - `python -m markitdown output.pptx`
  - `soffice --headless --convert-to pdf output.pptx`
- 若解析/转换失败，禁止直接交付，必须返工

### C. 设计规则
- 禁止整套 PPT 全是纯 bullet 页
- 禁止所有页面同一布局
- 禁止大段文字直接贴上去
- 禁止默认蓝白政府风
- 禁止“标题下面一条横线”的廉价 AI 感布局

### D. 高级感规则
每套 deck 至少具备：
- 1 页强封面
- 1 页管理摘要 / 关键结论页
- 1 页流程 / 架构 / 对比 / 卡片类结构化页面
- 视觉主色、辅助色、强调色明确分工
- 页面之间有统一视觉 motif（卡片、标题条、图形语言、图标语言之一）

## 标准工作流（必须严格执行）

### Step 1：明确输入
先确定以下信息：
- 主题 / 题目
- 受众（老板/客户/管理层/内部培训/投标）
- 目标（汇报/售前/宣讲/路演/复盘）
- 输入源（Word/PDF/MD/TXT/飞书文档/纪要/用户口述）
- 语言（中文/中英双语）
- 页数范围（默认 8~15 页）
- 是否有模板 / 品牌色 / Logo

如果缺失但不影响主链路，可先用默认值：
- 受众：管理层
- 目标：管理汇报
- 页数：10~12 页
- 风格：科技感 + 极简 + 管理汇报气质

### Step 2：先做“结构化策划”，再做 PPT
在生成 `.pptx` 之前，先形成 slide plan：
- 封面
- 管理摘要
- 背景 / 问题
- 方案 / 架构 / 模块
- 流程 / 路线图
- 风险 / 价值 / 下一步

必须先把页面类型想清楚，再生成。

### Step 3：页面类型分配
至少混合使用以下页面类型中的 3 类：
- summary（摘要卡片）
- problem（痛点/挑战）
- comparison（现状 vs 目标）
- architecture（架构图 / 集成边界）
- scope（模块/范围卡片）
- process（流程 4 步 / 5 步）
- timeline（里程碑）
- metrics（指标收益）
- risk（风险依赖）
- next-step（行动建议）

### Step 4：生成 `.pptx`
优先调用 `PPT-GEN`。
如需本地精修或模板级修补，再结合 `powerpoint` 技能。

### Step 5：生成后 QA（必须）
生成后必须执行：

#### 结构 QA
至少一项：
```bash
python -m markitdown output.pptx
```
或
```bash
python scripts/office/soffice.py --headless --convert-to pdf output.pptx
```

#### 视觉 QA
尽量做：
1. 转 PDF / 图片
2. 检查是否有：
   - 文本溢出
   - 元素重叠
   - 颜色冲突
   - 标题过长换行难看
   - 页间风格不统一
   - 留白不足

若发现问题，必须修正后再交付。

## 默认审美标准（对标 Kimi 质量）

### 1. 风格关键词
默认风格：
- 科技感
- 极简
- 管理汇报感
- 高留白
- 专业咨询气质

### 2. 颜色规则
默认采用：
- 主色：深蓝 / 深灰 / 深墨色
- 辅色：低饱和浅灰蓝 / 冰蓝 / 白
- 强调色：橙、青、绿色三选一

要求：
- 一主一辅一强调，不要彩虹色
- 全文视觉由主色控制，不要每页换色系

### 3. 版式规则
- 封面允许暗底大字
- 内容页优先浅底深字
- 摘要页优先卡片化
- 架构/流程页优先图形化
- 一页不超过 5 个核心 bullet
- 标题要结论化，不要“项目介绍”“方案概述”这种空标题

### 4. 文案规则
把普通标题改成管理表达：
- 差：项目背景
- 好：当前建设窗口已形成，需统一推进 MRP 能力落地

- 差：系统架构
- 好：方案采用“主数据—计划—执行反馈”三层闭环架构

- 差：实施计划
- 好：建议按“三阶段推进”，先试点再复制推广

## 默认执行策略

### 有清晰资料源
- 用资料直接提炼
- 不主动编造内容
- 缺失部分用“建议页”方式补齐

### 只有一个主题，没有资料
- 先自动构造专业目录
- 再按目录生成内容
- 明确标注这是一版“结构化初稿”

### 用户强调高级感
- 提升视觉页占比
- 减少大段 bullet
- 加强封面、摘要、流程、架构页的设计力度

### 用户强调稳定打开
- 优先稳定模式
- 少做高风险 XML 拼装
- 先保证可打开性，再追求复杂视觉
- **如果用户已经反馈 PowerPoint/WPS 打开会提示“修复”，立即放弃当前高风险生成链路**（尤其是手写 OOXML、模板复制、复杂 shape 组合），切换到纯 `python-pptx` 安全生成
- 安全生成时优先只用：**矩形、文本框、基础直线**；尽量避免圆形、复杂几何、自动缩放、模板残留对象、手写 XML 关系
- 生成后必须做 `python-pptx` 读取 + 另存（roundtrip）验证；仅 `markitdown` 可解析还不够，桌面端兼容性要以 roundtrip 作为更强信号

## 失败兜底策略

如果高视觉版本出现兼容风险：
1. 降级为稳定布局版本
2. 保留配色、版式节奏、高级感骨架
3. 放弃高风险复杂元素
4. 优先保证 `.pptx` 正常打开

### 常见兼容性坑：XML 坐标出现浮点数
如果生成器输出的 OOXML 中出现 `x='1700784.0'`、`cx='5404104.0'` 这类浮点 EMU 值，`markitdown`/部分解析器可能直接报错，虽然 PowerPoint 有时还能勉强打开。

处理方式：
- 解包 `.pptx`
- 仅对 XML 中坐标类属性做整数化：`x/y/cx/cy/w/h/id/rot/lIns/tIns/rIns/bIns`
- 用 `int(round(float(value)))` 回写
- 重新打包后再次执行 `markitdown` 校验

### 常见兼容性坑：同一页内重复 shape id
如果幻灯片 XML 中多个 `<p:cNvPr id="...">` 重复使用同一个 id（常见于手写 XML 拼装），PowerPoint/WPS 很容易在打开时提示“发现问题并已修复”。

处理方式：
- 对每一页单独扫描所有 `<p:cNvPr ... id="N" ...>`
- 按出现顺序重排为唯一整数 id（例如从 1000 开始递增）
- 重新打包后再做 `markitdown` 校验
- 最好再用 `python-pptx` 做一次读取/另存验证

### 关键降级策略：反复报“修复”时，立刻放弃高风险模板/XML链路
如果已经出现以下情况之一：
- PowerPoint / WPS 连续两版仍提示“发现问题并已修复”
- `markitdown` 能解析，但电脑端仍报修复
- 已经修过浮点坐标、重复 shape id，桌面端问题依旧存在

不要继续在 XML 拼装链路上反复补丁，应立即切换到 **纯 `python-pptx` 安全生成**。

#### 安全生成要求
- 只使用低风险元素：`Rectangle`、`TextBox`、基础 `Line/Connector`
- 少用或避免：复杂自定义 XML、模板复制、手工关系文件改写、过多特殊造型
- 页面设计靠：留白、层级、色块、字号、卡片化，而不是复杂对象
- 先做稳定版，再逐步加高级感

#### 稳定版强制验证
切到纯 `python-pptx` 后，至少做 3 项验证：
1. `python -m markitdown output.pptx`
2. `python - <<'PY' ... Presentation(output).save(roundtrip.pptx) ... PY`
3. 扫描每页 `cNvPr id` 是否重复

如果 roundtrip 保存成功，通常比仅靠 `markitdown` 更接近真实桌面端兼容性，但**仍不能替代用户在电脑端 PowerPoint / WPS 的实际打开确认**。

#### 最终验收口径（必须）
自动化校验只能证明“结构风险较低”，**不能直接等同于“桌面端一定不报修复”**。

因此交付口径必须遵守：
- 可以说“已通过结构校验 / roundtrip 校验”
- **不能在未经过用户电脑端实测前，宣称“PowerPoint/WPS 一定不报修复”**
- 如果用户反馈电脑端仍报修复，立即判定当前链路不合格，继续降级或重做

#### 审美补偿原则
即使走稳定版，也不要退化成普通白底 bullet：
- 封面必须有强主视觉和结论化标题
- 摘要页必须卡片化
- 至少保留一页对比/流程/路线图
- 文案改成管理表达：结论、判断、建议、拍板事项

原则：
**宁可牺牲一点花哨，也不能交付一个会报修复或无法解析的 PPT。**

- `python-pptx` 可读取
- 但用户在 PowerPoint / WPS 电脑端打开仍提示“修复”

则不要继续在高风险 XML 模板链路上反复打补丁，应直接切换到 **纯 `python-pptx` 安全生成链路**：
- 不手写 slide XML
- 不复制复杂高风险模板结构
- 用原生 shape / textbox / connector 重建页面
- 先保证“实际打开不修复”，再逐步追回高级感

推荐验证顺序：
1. `python -m markitdown output.pptx`
2. `python -c 'from pptx import Presentation; Presentation("output.pptx").save("roundtrip.pptx")'`
3. 让用户在 PowerPoint/WPS 电脑端实际打开确认

如果第 3 步失败，默认判定当前链路不合格，切换到稳定优先方案。
