# 流程-数据-IT-治理一体化模型

## 目标
帮助 consulting agent 在回答复杂企业数字化问题时，避免只从单一维度出发，而是用统一模型做系统判断。

## 一、四层一体化结构

### 1. 流程层
关注：端到端流程是否完整，节点是否顺畅，控制点是否前置。
关键词：断点、重复审批、重复录入、例外流程、交接责任。

### 2. 数据层
关注：主数据是否统一，交易数据是否可追溯，指标口径是否一致。
关键词：多版本、对不上、口径冲突、分析不可信。

### 3. IT层
关注：系统边界是否清晰，主系统是否唯一，接口与状态回写是否稳定。
关键词：重叠建设、主系统冲突、集成断裂、状态不同步。

### 4. 治理层
关注：谁负责、谁拍板、谁维护、谁验收、谁持续运营。
关键词：没人负责、项目推进难、标准推不动、上线后失控。

## 二、常见错误
### 只看流程，不看数据
结果：流程改了，报表还是乱。

### 只看系统，不看治理
结果：系统上了，但部门不按新规则执行。

### 只看数据，不看流程
结果：BI 很漂亮，但基础动作没变。

### 只看项目计划，不看组织现实
结果：路线图很好看，但执行不断延后。

## 三、默认分析顺序
1. 业务目标是什么
2. 流程断点在哪
3. 数据断点在哪
4. 系统边界是否合理
5. 治理责任是否明确
6. 一期怎么切

## 四、判断一个方案是否靠谱的标准
一个方案如果要算靠谱，至少要同时回答：
- 业务为什么要这么做
- 流程怎么改
- 数据如何统一
- 系统怎么分工
- 谁来推进和负责

如果只回答了其中 1-2 层，通常都不够。

## 五、适合输出的场景
- 企业信息化蓝图
- 系统边界规划
- 数据治理方案
- 需求池优先级判断
- 项目推进策略
- 多系统改造路线图
